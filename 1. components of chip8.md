16 Bit Index Register
special register used to store memory addresses for use in operation
16 bit register because the maximum memory address 0xFFF is too big for an 8 bit register
For example, storing a 0xFFF value in a 16 bit register
0xFFF in binary: 1111 1111 1111 (12 bits)
Inside a 16-bit register: 0000 1111 1111 1111
an 8 bit register would not be able to store all 12 bits.

16 Bit Program Counter
CPU needs a way to keep track of which instruction to excute next from address 0x2000 onwards
PC is a special register that holds the address of next instruction to execute
again, 16 bit PC because the maximum memory address 0xFFF is too big for an 8 bit PC
An instruction is two bytes (16 bit), but memory is addressed as a single byte, 
so when we fetch an instruction from memory we need to fetch a byte from PC, and a byte from PC+1 and connect them into 1 value
then increment PC by 2.
increment PC before executing any instructions because some instructions will manipulate the PC to control program flow.
(fundamental rule of the Fetch-Decode-Execute cycle)

16 Level Stack
A stack is a way for CPU to keep track of the order of execution when it calls into functions
there is an instruction CALL that will cause CPU to begin executing instructions in a different region of the program
when the program reaches another instruction RET
it must be able to reutnr to where it was when it hit the call inst
the stack holds the PC value when the call instruction was executed, and the return statement pulls that address from the stack and put it back into the PC, so the CPU will execute it on next cycle

16 levels of stack means it can hold up to 16 different PCs
multiple levels allow for one function to call another function and so on until they all return to original caller site
17th call will cause stack overflow
Putting a PC onto the stack is called pushing and pulling a PC off of the stack is called popping.

8-bit Stack Pointer
we need a stack pointer to tell use where in the 16 level of stack our most recent value was placed
we only need 8 bits for our stack pointer because the stack will be represented as an array
so our stack pointer can just be an index into that array
we only need sixteen indices, which a single byte can manage
when we pop a value off the stack we dont delete from the array, but instead just copy the value and decrement SP 
Stack grows downward

Solution and Explanation:
CPU does fetch → increment → execute, in that order:
1. Read the instruction at PC
2. PC += 2 (every instruction is 2 bytes, hence $200, $202, $204…)
3. Then run it

by the time CALL actually executes, PC already points at the following instruction. Hence at CALL $208 at $200 pushes $202 into the stack because PC+2.
Nuances to take note:
CALL addr: push PC+2 into stack, SP++; PC = addr; // CALL will store PC+2 and push into stack, increment SP, and current PC is the instruction's address
RET: SP--; PC = stack[SP]; // discards PC+2 and reduce SP, PC is the popped stack
JMP addr: PC = addr; never touches the stack. // discards PC+2 and reduce SP, PC is the address

64x32 Monochrome Display Memory
CHIP-8 has an additional memory buffer used for storing the graphics to display
it is 64 wide x 32 high, each pixel is either on or off, so 2 colours representation
we’ll use a uint32 for each pixel
on is 0xFFFFFFFF and off is 0x00000000
draw instruction
iterates over each pixel in a sprite and XORs the sprite pixel with the display pixel.
Sprite Pixel Off XOR Display Pixel Off = Display Pixel Off
Sprite Pixel Off XOR Display Pixel On = Display Pixel On
Sprite Pixel On XOR Display Pixel Off = Display Pixel On
Sprite Pixel On XOR Display Pixel On = Display Pixel Off

Sprite pixel = a bit in sprite data sitting in RAM. Read-only during the draw. It's the rubber stamp.
Display pixel = one of the 2048 entries in your video framebuffer. Mutable. It's the paper.

DXYN reads the stamp and modifies the paper. That's the only relationship between them. Two names because they're two different memory regions playing two different roles — source and destination.

The key property: CHIP-8 never clears the screen between frames. There is no "redraw everything each tick." Whatever you drew stays there until CLS wipes it or another XOR flips it. The framebuffer is the state.
That's why XOR is the whole design. XOR is self-inverse — stamp the same sprite at the same spot twice and you're back where you started. So the move-a-sprite routine is: draw at (X,Y) to erase, draw at (X+1,Y+1) to appear. Two draws per movement, and if your display refreshes between them you catch a frame where the sprite is gone. That's the flicker.
In code it collapses to one line, because XOR against all-ones is a bitwise NOT:
cif (sprite_bit) {
    if (video[idx] == 0xFFFFFFFF) registers[0xF] = 1;   // collision
    video[idx] ^= 0xFFFFFFFF;                            // flip on↔off
}

uint32_t video[64 * 32]{}
Why uint32_t for one bit of information? Purely an SDL convenience, and the tutorial says so. SDL's ARGB8888 texture format wants 4 bytes per pixel. If your buffer already stores 0xFFFFFFFF/0x00000000, you hand SDL the array pointer directly:
SDL_UpdateTexture(texture, nullptr, video, 64 * sizeof(uint32_t));
It's the whole reason on is 0xFFFFFFFF and not 1.



uint16_t opcode
16 bits because an instruction is exactly 2 bytes. 
opcode is a scratch variable holding the one instruction currently being executed, overwritten every cycle:
opcode = (mem[pc] << 8) | mem[pc+1];   // fetch → opcode
pc += 2;
switch (opcode & 0xF000) { ... }        // decode + execute → opcode is now dead
